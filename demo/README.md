# Demo video production

The project's hero video at `demo/output/demo.mp4` is generated by a
scripted pipeline. You should not re-record it manually with a screen
recorder. Follow the steps below.

## One-time setup

```bash
brew install vhs ffmpeg           # vhs drives the terminal clips
npx playwright install chromium   # Playwright drives the browser clips
```

`vhs` requires `ttyd` on PATH (Homebrew installs it as a dependency).

## Making a video

### 1. Regular update (UI hasn't changed meaningfully)

```bash
npm run demo:record
open demo/output/demo.mp4
```

Watch it. Go through the checklist in §4. If everything passes, ship.

### 2. Updating after a UI change

The test uses a handful of user-facing selectors. If the UI changed, one
of these may need an update in `demo/demo.spec.ts`:

| What changed in the UI                     | Fix                                            |
| ------------------------------------------ | ---------------------------------------------- |
| Button label ("View diff", "Send & finish") | Update the `getByRole({ name: ... })` call     |
| Banner got a new testid                    | Update `getByTestId('review-banner')`          |
| Comment form data attribute renamed        | Update `[data-comment-form]` CSS + selectors   |
| Doc body copy changed                      | Update `demo/fixtures/sample-before.md`        |
| CLI output copy should reflect new UI      | Update `demo/mock-claude.sh`                   |

Re-run `npm run demo:record`, then walk the checklist in §4.

### 3. Adding a new scene (new feature)

1. Sketch what the new scene shows and how long (seconds), then fit it
   into the total 30-32s budget — trim existing scenes to make room.
2. If it's a new CLI scene, add a new `.tape` file under `demo/` and a
   matching mode to `demo/mock-claude.sh`. If it's a new MDR scene, add
   a new `startScreenCapture('clip-XX')` block in `demo/demo.spec.ts`.
3. Wire the new clip into `demo/record.sh` alongside the existing
   `clip-01` through `clip-04` stitching + xfade chain.
4. Run, watch, walk §4.

## 3. Tuning pacing

All timing knobs live in **one `T` constants block** at the top of
`demo/demo.spec.ts`. Named `MDR_INITIAL_DWELL`, `DRAG_DURATION`,
`SUBMIT_RIPPLE_DWELL`, etc. Edit there, don't scatter magic numbers.

CLI-side timings live in the `.tape` files (`demo/demo-terminal-1.tape`,
`demo/demo-terminal-2.tape`) — typing speed, clip-01 tail hold, clip-03
tail hold. The header comment in `demo.spec.ts` explains which rule
each tape timing enforces.

The **settled decisions** at the top of `demo/demo.spec.ts` explain why
certain things are NOT knobs (e.g. zoom is off on purpose). Read that
comment block before re-enabling anything that looks dormant.

## 4. Quality checklist before shipping

Watch the final video end-to-end and confirm:

- [ ] CLI types fit on one line (no mid-word wraps).
- [ ] First MDR frame dwells ≥2s before anything moves (the "ta-da").
- [ ] Drag-select: cursor tip visually lands ON the anchor text, not a
      line above/below. Selection highlight tracks the cursor.
- [ ] Comment form appears right under the highlighted text.
- [ ] "Comment" submit: blue ripple is clearly visible ON the button
      before the form disappears.
- [ ] No visible pan or drift during any transition.
- [ ] Drag handles (if visible) align with the comment anchor.
- [ ] CLI→MDR transitions: clicked element is on screen for ≥1s before
      the xfade begins.
- [ ] Clip-03: "Both comments addressed..." visible ~1.3s before the
      cut back to MDR. Matches the clip-01 "Waiting for review..." tail.
- [ ] Final frame is clean — no tooltips, no hover states.
- [ ] Total duration in the 30-32s range.

```bash
# quick duration check
ffprobe -v error -show_entries format=duration -of csv=p=0 demo/output/demo.mp4
```

## 5. Shipping

```bash
git add demo/output/demo.mp4
git commit -m "demo: refresh video"
# Upload demo/output/demo.mp4 to the GitHub release asset host or
# wherever the README currently points.
# Update the link in README.md if the URL changed.
```

## 6. Troubleshooting

**"Request too large" during development iteration.** Not about the
video — that's a conversation-context limit when pasting lots of frame
screenshots. Extract frames one at a time with `ffmpeg -ss T -frames:v 1`
rather than grabbing a whole sequence.

**VHS fails with "failed to execute command".** Flaky ttyd connection.
Just re-run `npm run demo:record`. It usually passes on retry.

**Playwright test times out.** Usually the selector the test clicks has
been renamed. See the table in §2.

**Video too short / too fast.** The PNG→JPEG capture switch was made
because PNG screenshots blocked the test's event loop and silently
compressed the apparent timeline. If you ever revert to PNG, expect
~20% compression and plan accordingly. Keep it JPEG.

**Video too long.** Trim in this order:
1. `T.FINAL_HOLD` in clip-04 (each 100ms shaves directly)
2. `T.POST_COMMENT_HOLD` between comments
3. Tape `Sleep` values (clip-01 and clip-03 tails)
4. `T.CURSOR_TO_*` slowMove durations

Floors you shouldn't cross: `DRAG_DURATION ≥ 600`, `SUBMIT_RIPPLE_DWELL ≥ 440`,
`POST_SEND_HOLD ≥ 1300`, `TYPE_DELAY ≥ 14`. Going under these breaks
the quality checks in §4.
